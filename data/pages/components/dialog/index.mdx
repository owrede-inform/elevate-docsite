## Overview


### When to Use Dialogs
- **Critical decisions**: Destructive actions, important confirmations
- **Required input**: Essential information needed to continue
- **Error handling**: Critical errors that need immediate attention
- **Complex forms**: Multi-step processes or detailed data entry
- **Content preview**: Quick view of detailed information

### When NOT to Use Dialogs
- **Simple notifications**: Use Toast or Notification components
- **Non-critical information**: Use inline messaging or cards
- **Primary navigation**: Use appropriate navigation patterns
- **Frequent interactions**: Consider inline editing or side panels

## Design Principles


Understanding these core principles will help you make better design decisions and avoid interface patterns that confuse or frustrate users.

### Interruption Management Principle
**Dialogs should interrupt workflow only when absolutely necessary.** Overusing dialogs creates friction and trains users to dismiss them without reading. Reserve dialogs for truly important decisions or required input.

### Focus Containment Principle  
**User attention should be fully contained within the dialog.** The background should be obscured and interaction blocked to prevent confusion. Users need clear entry and exit points from the modal state.

### Progressive Disclosure Principle
**Complex information should be revealed gradually.** Long forms or detailed content in dialogs creates overwhelming experiences. Break complex interactions into manageable steps or consider alternative patterns.

### Clear Intent Principle
**Every dialog should have a clear purpose and outcome.** Users should understand why the dialog appeared and what they need to do. Ambiguous dialogs create anxiety and decision paralysis.

### Key Properties

The most commonly used properties for design and development:

<PageTable>

| Property | Options | Description |
|----------|---------|-------------|
| `open` | `true`, `false` (default) | Whether the dialog is visible |
| `size` | `small`, `medium` (default), `large`, `xl` | Dialog width and height |
| `full-screen` | `true`, `false` (default) | Whether dialog fills entire viewport |
| `persistent` | `true`, `false` (default) | Whether dialog can be closed by clicking overlay |
| `modal` | `true` (default), `false` | Whether dialog blocks background interaction |

</PageTable>

**Dialog Structure:** Use `elvt-dialog-header`, `elvt-dialog-content`, and `elvt-dialog-actions` for consistent layout.

## Accessibility


### Essential Accessibility Practices

**Focus management is automatic:**
```html
<!-- Good: Focus moves to dialog and is trapped -->  
<elvt-dialog open="true">
  <elvt-dialog-header>
    <h2>Dialog Title</h2> <!-- Receives initial focus -->
  </elvt-dialog-header>
  <elvt-dialog-content>
    <p>Dialog content</p>
  </elvt-dialog-content>
  <elvt-dialog-actions>
    <elvt-button>Cancel</elvt-button>
    <elvt-button tone="primary">Confirm</elvt-button>
  </elvt-dialog-actions>
</elvt-dialog>
```

**Provide descriptive titles:**
- ✅ "Delete 3 selected items" 
- ⛔️ "Confirm" or "Dialog"

**Use appropriate dialog roles:**
- Alert dialogs: `role="alertdialog"` (automatically applied)
- Standard dialogs: `role="dialog"` (automatically applied)
- Form dialogs: Use proper form labels and structure

**Keyboard behavior is built-in:**
- Escape key closes dialog (unless `persistent="true"`)
- Tab navigation is trapped within dialog
- Focus returns to trigger element when closed

**Screen reader announcements:**
- Dialog title is announced when opened
- Content changes are communicated appropriately
- Error states in forms are announced

### Critical Requirements

- Always provide clear, descriptive dialog titles
- Ensure all interactive elements are keyboard accessible
- Don't nest dialogs (avoid opening dialogs from other dialogs)
- Provide escape mechanisms (close button or cancel action)
- Use semantic HTML structure within dialog content
- Test with screen readers and keyboard-only navigation
