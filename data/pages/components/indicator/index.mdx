## Overview

### When to Use Indicators
- **Status communication**: Show current state of processes, connections, or system health
- **Notification badges**: Display count of unread messages, alerts, or pending items
- **Progress feedback**: Indicate completion status of tasks or loading states
- **Availability states**: Show online/offline status or resource availability
- **Validation feedback**: Provide immediate response to user input validation

### When NOT to Use Indicators
- **Complex information**: Use detailed messages for complex status explanations
- **Static content**: Don't use indicators for unchanging information
- **Primary navigation**: Indicators should supplement, not replace, clear labeling
- **Critical actions**: Don't rely solely on indicators for important confirmations

## Design Principles

### Clarity Principle
**Indicators should communicate status immediately and unambiguously.** Users should understand the meaning of an indicator at first glance without needing additional explanation or context.

### Consistency Principle  
**Similar states should use consistent visual treatment across the interface.** Error states should always use the same color and style, success states should be consistent, and so on.

### Hierarchy Principle
**Different urgency levels should be visually distinct and appropriately prioritized.** Critical alerts should be more prominent than informational messages.

### Non-Interference Principle
**Indicators should inform without disrupting the primary user workflow.** They should provide status information without blocking or distracting from the main task.

## Best Practices

### Choosing Appropriate Indicator Types

**Why this matters:** Different situations require different types of visual feedback. Badge counts work well for notifications, while progress bars are better for showing completion states.

### Using Color Meaningfully

**Why this matters:** Color carries semantic meaning in interface design. Users expect red to indicate danger or errors, green for success, and yellow for warnings or cautions.

### Providing Alternative Information

**Why this matters:** Not all users can distinguish colors or see indicators clearly. Providing text labels, icons, and alternative cues ensures accessibility for all users.

### Managing Indicator Density

**Why this matters:** Too many indicators can overwhelm users and reduce the effectiveness of important status information. Strategic placement and prioritization maintain clarity.

## Accessibility

### Essential Accessibility Practices

**Color independence:**
- Never rely solely on color to communicate status or meaning
- Provide text labels, icons, or patterns alongside color indicators
- Ensure sufficient color contrast for all indicator states
- Test indicators with color blindness simulation tools

**Screen reader support:**
- Use appropriate ARIA labels to describe indicator states
- Ensure status changes are announced to screen readers
- Provide alternative text for icon-only indicators
- Use live regions for dynamic status updates

**Keyboard accessibility:**
- Interactive indicators should be keyboard accessible
- Provide clear focus indicators for actionable elements  
- Ensure tab order makes sense with indicator placement
- Allow keyboard users to access indicator information

**Clear communication:**
- Use familiar icons and symbols that have established meanings
- Provide hover tooltips or expandable descriptions for complex indicators
- Keep indicator text concise but descriptive
- Test indicator comprehension with diverse user groups
