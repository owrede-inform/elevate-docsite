## Overview


### When to Use Cards
- **Content grouping**: Related information that belongs together logically
- **Data display**: Metrics, statistics, dashboards, or summary information
- **Navigation**: Category links, feature highlights, landing page sections
- **Product listings**: Items in catalogs, directories, or search results
- **Information hierarchy**: Breaking complex content into digestible sections

### When NOT to Use Cards
- **Simple lists**: Use List component for basic item collections
- **Form sections**: Use proper form fieldsets or sections for form organization
- **Full-page layouts**: Use appropriate layout components for main page structure
- **Inline content**: Use appropriate text components for paragraph-level information

## Design Principles


Understanding these core principles will help you create card layouts that enhance rather than complicate information consumption and user task completion.

### Information Hierarchy Principle
**Cards should create clear visual hierarchy within content.** Users need to quickly scan and prioritize information within cards. Poor hierarchy forces users to read everything to find what they need, slowing down task completion.

### Content Cohesion Principle  
**Related information should be grouped together logically.** Cards work best when they contain information that naturally belongs together. Mixing unrelated content confuses users and breaks their mental models of how information is organized.

### Visual Consistency Principle
**Similar content types should use similar card treatments.** Consistent card patterns help users predict where to find information and how to interact with content. Visual inconsistency creates cognitive overhead and reduces interface familiarity.

### Contextual Appropriateness Principle
**Card treatment should match the content's importance and purpose.** High-priority content may need elevated cards, while background information works better with subtle treatments. Mismatched visual weight creates confusion about content importance.

## Implementation


### Installation

```bash
npm install @inform-elevate/elevate-core-ui
```

### Basic Usage

```html
<elvt-card>
  <h3>Card Title</h3>
  <p>Card content goes here with relevant information and details.</p>
</elvt-card>
```

### With Header and Footer

```html
<elvt-card>
  <div slot="header">
    <h3>Card Title</h3>
    <elvt-button tone="subtle" size="small">Action</elvt-button>
  </div>
  
  <p>Main card content with detailed information.</p>
  
  <div slot="footer">
    <elvt-button tone="primary">Primary Action</elvt-button>
    <elvt-button tone="secondary">Secondary</elvt-button>
  </div>
</elvt-card>
```

### Different Border Styles

```html
<elvt-card border="rounded">
  <p>Card with rounded corners only</p>
</elvt-card>

<elvt-card border="none">
  <p>Borderless card</p>
</elvt-card>
```

### Layer Variations

```html
<elvt-card layer="elevated">
  <h3>Elevated Card</h3>
  <p>This card appears raised with shadow.</p>
</elvt-card>

<elvt-card layer="overlay">
  <h3>Overlay Card</h3>
  <p>Card designed to overlay other content.</p>
</elvt-card>
```

### With Custom Padding

```html
<elvt-card padding="xl">
  <h3>Large Padding Card</h3>
  <p>This card has extra large padding.</p>
</elvt-card>

<elvt-card padding="sm md">
  <p>Card with small vertical, medium horizontal padding.</p>
</elvt-card>
```

### Key Properties

The most commonly used properties for design and development:

<PageTable>

| Property | Options | Description |
|----------|---------|-------------|
| `border` | `all` (default), `rounded`, `none` | Border treatment for different visual contexts |
| `layer` | `default`, `elevated`, `overlay`, `sunken` | Visual depth and prominence level |
| `padding` | Design token values | Custom spacing using consistent design tokens |

</PageTable>

### Slots

<PageTable>

| Slot | Description |
|------|-------------|
| `default` | Main card content area |
| `header` | Header content (automatically detected when used) |
| `footer` | Footer content (automatically detected when used) |

</PageTable>

*For complete API reference including all properties, events, and methods, see the [Storybook documentation](https://elevate-core-ui.inform-cloud.io/?path=/docs/components-card--docs).*

## Accessibility

### Semantic Structure
- Cards use appropriate landmark roles when needed
- Interactive cards have proper link or button semantics
- Content hierarchy is maintained with proper heading levels

### Keyboard Support
- **Tab**: Moves focus to interactive elements within the card
- **Enter/Space**: Activates interactive cards
- **Arrow keys**: Navigate between cards in a grid (when implemented)

### Screen Reader Support
- Card content is announced in logical reading order
- Interactive cards have appropriate accessible names
- Related content is grouped appropriately

### Focus Management
- Clear focus indicators on interactive elements
- Focus moves logically through card content
- Focus is trapped appropriately in modal card contexts

### Best Practices
- Ensure sufficient color contrast for all text and borders
- Provide alternative text for images within cards
- Use semantic heading hierarchy within cards
- Make interactive areas sufficiently large (44px minimum)
- Don't nest interactive elements within interactive cards
