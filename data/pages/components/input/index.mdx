## Overview

### When to Use Inputs
- **Form fields**: Registration, login, profile editing, and settings
- **Search functionality**: Site search, filtering, and data lookup
- **Data entry**: Numeric values, dates, measurements, and calculations
- **User feedback**: Comments, reviews, messages, and notes

### When NOT to Use Inputs
- **Multi-line text**: Use Textarea component for longer content
- **Selection from options**: Use Select, Radio, or Checkbox components
- **File uploads**: Use File Dropzone component instead
- **Toggle states**: Use Switch component for on/off controls

## Design Principles

### Clarity Principle
**Input purpose should be immediately clear from labels and context.** Users should never have to guess what information to enter or what format is expected. Ambiguous inputs increase errors and frustration.

### Progressive Enhancement Principle  
**Start with basic functionality and add enhancements strategically.** Every feature added to an input should serve a clear user need. Over-engineered inputs create confusion and maintenance burden.

### Validation Timing Principle
**Provide feedback when it's most helpful, not intrusive.** Real-time validation should guide users toward success, not constantly interrupt their input flow with premature error messages.

### Consistency Principle
**Similar inputs should behave similarly across the application.** Inconsistent validation, styling, or behavior patterns force users to relearn interactions and increase cognitive load.

## Best Practices

```javascript
// Import all components

import '@inform-elevate/elevate-core-ui/dist/components/input/input.js';
```

### Common Usage Patterns

```html
<!-- Basic text input -->
<elvt-input type="text" placeholder="Enter your name">
  <label slot="label">Full Name</label>
</elvt-input>

<!-- Email input with validation -->
<elvt-input type="email" required placeholder="name@company.com">
  <label slot="label">Email Address</label>
  <span slot="help-text">We'll never share your email address</span>
</elvt-input>

<!-- Search input with icon and clear button -->
<elvt-input type="search" clearable placeholder="Search products...">
  <label slot="label">Product Search</label>
  <elvt-icon slot="prefix" icon="mdi:magnify"></elvt-icon>
</elvt-input>

<!-- Password input with requirements -->
<elvt-input type="password" required minlength="8">
  <label slot="label">Password</label>
  <span slot="help-text">Must be at least 8 characters with one number</span>
</elvt-input>

<!-- Number input with constraints -->
<elvt-input type="number" min="18" max="100" step="1">
  <label slot="label">Age</label>
</elvt-input>

<!-- Currency input with prefix -->
<elvt-input type="number" step="0.01" min="0">
  <label slot="label">Price</label>
  <span slot="prefix">$</span>
</elvt-input>
```

### Key Properties

The most commonly used properties for design and development:

<PageTable>

| Property | Options | Description |
|----------|---------|-------------|
| `type` | `text` (default), `email`, `password`, `number`, `search`, `tel`, `url`, `date`, `time`, `datetime-local` | Input type that determines keyboard and validation behavior |
| `size` | `small`, `medium` (default), `large` | Physical size of the input field |
| `placeholder` | string | Placeholder text shown when input is empty |
| `required` | `true`, `false` (default) | Whether the input must be filled before form submission |
| `disabled` | `true`, `false` (default) | Whether the input can be interacted with |
| `clearable` | `true`, `false` (default) | Show clear button when input has a value |
| `readonly` | `true`, `false` (default) | Whether the input value can be modified |
| `autocomplete` | string | Automated assistance for filling out field values |
| `maxlength` | number | Maximum number of characters allowed |
| `minlength` | number | Minimum number of characters required |
| `pattern` | string | Regular expression pattern for validation |
| `min` | number/string | Minimum value for number/date inputs |
| `max` | number/string | Maximum value for number/date inputs |
| `step` | number | Step interval for number inputs |

</PageTable>

**Slots:** Use `slot="prefix"` for icons before input, `slot="suffix"` for icons after input. Labels and help text should be handled by wrapper components or field containers.

*For complete API reference including all properties, events, and methods, see the [Storybook documentation](https://elevate-core-ui.inform-cloud.io/?path=/docs/components-forms-input--docs).*

## Accessibility

### Keyboard Support
- **Tab**: Moves focus to the input
- **Shift + Tab**: Moves focus away from the input
- **Arrow keys**: Navigate within text (standard text editing)
- **Enter**: Submits form (if applicable)

### Screen Reader Support
- Labels are properly associated with inputs using `for` attribute
- Required fields are announced to screen readers
- Validation messages are announced when they appear
- Helper text is associated with the input

### Focus Management
- Clear focus indicators that meet WCAG contrast requirements
- Focus remains on input during validation
- Focus moves logically through form fields

### Best Practices
- Always provide labels for all inputs
- Use appropriate input types for better mobile keyboards
- Ensure error messages are descriptive and actionable
- Maintain sufficient color contrast (4.5:1 minimum)
- Don't rely solely on color to indicate validation states
