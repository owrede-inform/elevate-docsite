## Overview

### When to Use Fields
- **Form creation**: Wrap input elements with consistent labeling and help text
- **Validation display**: Show error states and success feedback to users
- **Required indicators**: Mark mandatory fields with visual indicators
- **Help text**: Provide additional context and guidance for complex inputs
- **Consistent spacing**: Maintain uniform field layout throughout forms

### When NOT to Use Fields
- **Simple inputs**: Basic inputs without labels or validation may not need field wrapper
- **Custom layouts**: Highly specialized form layouts might require custom field structures
- **Non-form elements**: Don't wrap non-input elements in field components

## Design Principles

### Consistency Principle
**Form fields should maintain consistent layout, spacing, and visual hierarchy.** Consistent field patterns create predictable interfaces that users can navigate efficiently.

### Clarity Principle  
**Labels and help text should be clear, concise, and actionable.** Ambiguous field labels create confusion and increase form abandonment rates.

### Feedback Principle
**Validation states should provide immediate, helpful feedback to users.** Clear error messages guide users toward successful form completion.

### Accessibility Principle
**All form fields must be keyboard accessible and screen reader friendly.** Proper labeling and focus management ensure inclusive form experiences.

## Best Practices

### Writing Effective Labels

**Why this matters:** Clear labels reduce cognitive load and help users complete forms successfully. Vague or technical labels increase abandonment rates and user frustration.

### Providing Helpful Validation

**Why this matters:** Good validation prevents errors before they happen and guides users toward success. Generic error messages frustrate users and don't help them fix problems.

### Managing Field States

**Why this matters:** Proper state management provides clear feedback about field status and helps users understand form progress and requirements.

### Basic Field Usage

```html
<!-- Basic field with label -->
<elvt-field label="Email Address">
  <elvt-input type="email" placeholder="your@email.com"></elvt-input>
</elvt-field>

<!-- Required field -->
<elvt-field label="Password" required>
  <elvt-input type="password" required></elvt-input>
</elvt-field>

<!-- Field with help text -->
<elvt-field label="Username" help-text="Must be 3-20 characters">
  <elvt-input type="text" placeholder="your_username"></elvt-input>
</elvt-field>

<!-- Field with error state -->
<elvt-field label="Email" error="Please enter a valid email">
  <elvt-input type="email" error></elvt-input>
</elvt-field>

<!-- Field with success state -->
<elvt-field label="Username" success="Username is available">
  <elvt-input type="text" success></elvt-input>
</elvt-field>
```

### Field Attributes

<PageTable>

| Attribute | Type | Default | Description |
|-----------|------|---------|-------------|
| `label` | string | - | Field label text |
| `required` | boolean | false | Shows required indicator |
| `help-text` | string | - | Additional guidance text |
| `error` | string | - | Error message to display |
| `success` | string | - | Success message to display |
| `disabled` | boolean | false | Disables the entire field |

</PageTable>

### Key Components Used

Components commonly used with Field:

<PageTable>

| Component | Usage | Description |
|----------|---------|-------------|
| `elvt-input` | Text inputs | Standard text input field |
| `elvt-textarea` | Multi-line text | Large text input areas |
| `elvt-select` | Dropdown selection | Select from predefined options |
| `elvt-checkbox` | Boolean choices | Single or multiple selections |
| `elvt-radio` | Exclusive choices | Single selection from group |

</PageTable>

## Accessibility

### Essential Accessibility Features

**Proper labeling:**
- All fields have associated labels using proper semantic markup
- Required fields are clearly indicated both visually and for screen readers
- Help text is properly associated with inputs using aria-describedby
- Error messages are announced when validation fails

**Keyboard accessibility:**
- All fields are fully keyboard accessible with logical tab order
- Users can navigate between fields using Tab and Shift+Tab
- Input focus is clearly visible with appropriate focus indicators
- Submit and reset functions work with keyboard activation

**Screen reader support:**
- Field labels are properly announced when fields receive focus
- Help text and error messages are read aloud when relevant
- Field states (required, error, success) are communicated to assistive technology
- Form structure is clear with proper heading hierarchy

**Visual accessibility:**
- All text meets minimum contrast requirements for readability
- Error states don't rely solely on color to communicate issues
- Focus indicators are visible and meet accessibility guidelines
- Field sizing accommodates users who need larger text
