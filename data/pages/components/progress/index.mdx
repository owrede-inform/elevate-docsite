## Overview

### When to Use Progress
- **File uploads**: Show completion percentage for file upload operations
- **Data processing**: Indicate progress for long-running data operations
- **Multi-step workflows**: Track completion across sequential steps
- **Loading states**: Provide feedback during content loading or API calls
- **Installation processes**: Show progress for software installations or updates

### When NOT to Use Progress
- **Instant operations**: Don't show progress for operations that complete immediately
- **Unknown duration**: Use loading spinners for processes with unknown completion times
- **Simple page loading**: Basic page transitions may not need progress indicators

## Design Principles

### Clarity Principle
**Progress indicators should clearly communicate completion status and remaining work.** Ambiguous progress displays confuse users about system state and expected wait times.

### Feedback Principle  
**Progress should update frequently enough to feel responsive.** Too-infrequent updates make systems feel unresponsive, while too-frequent updates can be distracting.

### Context Principle
**Progress indicators should provide meaningful context about the operation.** Generic progress bars without explanation don't help users understand what's happening.

### Accessibility Principle
**Progress must be communicated to all users including those using assistive technology.** Screen readers need proper progress announcements and status updates.

## Best Practices

### Choosing Progress Types

**Why this matters:** The right progress type depends on whether completion can be measured. Determinate progress works for file uploads, while indeterminate progress suits unknown-duration operations.

### Providing Helpful Context

**Why this matters:** Context helps users understand what's happening and how long they might wait. "Loading..." is less helpful than "Processing 150 of 500 records."

### Managing User Expectations

**Why this matters:** Clear progress communication reduces user anxiety and abandonment. Users tolerate longer waits when they understand system status.

## Implementation

### Getting Started

```bash
yarn add @inform-elevate/elevate-core-ui
```

```javascript
// Import ELEVATE components

```

### Basic Progress Usage

```html
<!-- Basic determinate progress -->
<elvt-progress value="50" max="100" label="50% complete"></elvt-progress>

<!-- Indeterminate progress -->
<elvt-progress indeterminate label="Loading..."></elvt-progress>

<!-- Progress with variant -->
<elvt-progress value="75" max="100" variant="success" label="75% complete"></elvt-progress>

<!-- Progress with size -->
<elvt-progress value="60" max="100" size="large" label="60% complete"></elvt-progress>

<!-- Progress with custom max -->
<elvt-progress value="150" max="200" label="150 of 200 files processed"></elvt-progress>
```

### Progress Attributes

<PageTable>

| Attribute | Type | Default | Description |
|-----------|------|---------|-------------|
| `value` | number | 0 | Current progress value |
| `max` | number | 100 | Maximum progress value |
| `indeterminate` | boolean | false | Shows indeterminate progress animation |
| `variant` | string | primary | Color variant: primary, success, warning, error, info |
| `size` | string | medium | Size variant: small, medium, large |
| `label` | string | - | Accessible label for screen readers |
| `show-value` | boolean | false | Shows numerical value in progress bar |

</PageTable>

### JavaScript Progress Control

```javascript
// Get progress element
const progress = document.querySelector('elvt-progress');

// Update progress value
progress.value = 75;

// Switch to indeterminate
progress.indeterminate = true;

// Change variant
progress.variant = 'success';

// Listen for progress complete event
progress.addEventListener('progress-complete', (event) => {
  console.log('Progress completed:', event.detail);
});
```

## Accessibility

### Essential Accessibility Features

**Screen reader support:**
- Progress elements use proper ARIA roles and labels
- Progress updates are announced to screen readers
- Completion percentages are communicated clearly
- Context and status information is properly associated

**Keyboard accessibility:**
- Progress controls (pause, cancel) are keyboard accessible
- Focus management works properly during progress operations
- Users can access progress information without mouse interaction
- Tab order remains logical during progress updates

**Visual accessibility:**
- Progress bars meet minimum contrast requirements for all variants
- Progress indication doesn't rely solely on color coding
- Text labels provide clear status information
- Visual progress is supported by textual progress indicators

**Clear communication:**
- Progress labels describe what operation is in progress
- Time estimates or completion information is provided when possible
- Error states are clearly communicated with specific error messages
- Success states confirm completion with appropriate feedback
