## Overview


### When to Use Tables
- **Data comparison**: Comparing multiple items across different attributes
- **Structured datasets**: Financial data, inventory, user lists, and reports
- **Sortable information**: Content that benefits from multiple sort orders
- **Bulk operations**: Selecting multiple items for batch actions
- **Detailed listings**: Product catalogs, search results, and directory listings

### When NOT to Use Tables
- **Simple lists**: Use List component for basic item listings
- **Card layouts**: Use Card component for rich content with multiple data types
- **Mobile-first content**: Consider alternative layouts for small screens
- **Few data points**: Don't use tables for 2-3 columns of simple data

### Visual Styles

**Striped Rows** alternate row colors to improve scanning and readability.

**Bordered Tables** use borders to clearly separate cells and improve data clarity.

**Compact Tables** reduce spacing for dense data presentation.

**Card-Style Tables** use card-like styling for a softer, more modern appearance.

### Interactive Features

**Column Sorting** enables users to sort data by clicking column headers.

**Row Selection** provides checkboxes for selecting individual or multiple rows.

**Inline Editing** allows direct data editing within table cells.

**Column Filtering** includes filter controls for narrowing displayed data.

## Design Principles


### Scannability Principle
**Users should be able to quickly scan and compare data across rows and columns.** Visual design should support rapid information processing without creating fatigue or confusion.

### Data Integrity Principle  
**Information presentation should accurately reflect data relationships and hierarchies.** Misleading alignments, groupings, or visual treatments can cause users to misinterpret data.

### Progressive Complexity Principle
**Start with essential data, provide access to details when needed.** Tables should not overwhelm users with all available data but should make additional information accessible when relevant.

### Responsive Adaptation Principle
**Tables should remain functional across different screen sizes and interaction methods.** Large data tables need thoughtful responsive strategies that preserve usability on mobile devices.

## Best Practices


### Organizing Column Information

**Why this matters:** Poor column organization makes data comparison difficult and increases cognitive load. Logical column ordering and appropriate data types improve user efficiency.

### Managing Large Datasets

**Why this matters:** Large tables can overwhelm users and create performance issues. Proper pagination, filtering, and progressive loading maintain usability with extensive data.

### Providing Clear Data Hierarchy

**Why this matters:** Users need to understand data relationships and importance. Visual hierarchy through typography, spacing, and emphasis guides attention to key information.

### Key Properties

<PageTable>

| Property | Options | Description |
|----------|---------|-------------|
| `sortable` | `true`, `false` (default) | Enable column sorting functionality |
| `responsive` | `true` (default), `false` | Enable responsive table behavior |  
| `striped` | `true`, `false` (default) | Alternate row background colors for better scanning |
| `bordered` | `true`, `false` (default) | Add borders around table cells |
| `compact` | `true`, `false` (default) | Reduce padding for dense data presentation |
| `scroll-mode` | `virtual`, `normal` (default) | Enable virtual scrolling for large datasets |

</PageTable>

**Table Cell Properties:**
- `type`: `header` or `data` (default) - Defines cell type for accessibility
- `sortable`: Enable sorting for specific header columns
- `alignment`: `left` (default), `center`, `right` - Text alignment within cells

## Accessibility


### Essential Accessibility Practices

**Proper table structure:**
- Use semantic table elements (table, thead, tbody, th, td)
- Include table captions that describe table purpose
- Use proper heading hierarchy for table structure
- Associate data cells with appropriate headers

**Screen reader support:**
- Provide clear column headers with scope attributes
- Use aria-labels for complex table relationships
- Announce sorting states and selection changes
- Include summary information for large datasets

**Keyboard navigation:**
- Arrow keys navigate between table cells
- Tab moves through interactive elements within table
- Space/Enter activates buttons and checkboxes in cells
- Provide keyboard shortcuts for common table actions

**Visual accessibility:**
- Ensure sufficient contrast for table text and borders
- Use clear focus indicators for keyboard navigation
- Don't rely solely on color for data meaning
- Provide alternative text for data visualizations within tables
