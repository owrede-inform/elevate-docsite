# Button Component

<EsdsStatusBadge status="stable" />

The Button component provides consistent interactive elements for user actions. Built with accessibility in mind, it supports multiple visual styles, sizes, and states to handle various use cases throughout your application.

## Overview

Buttons are one of the most important interactive elements in any user interface. The ELEVATE Button component ensures consistent styling, behavior, and accessibility across your application while providing flexibility for different contexts.

## Basic Usage

The simplest button requires just a `tone` attribute to define its visual style:

<EsdsLiveExample>
  <elvt-button tone="primary">Click me</elvt-button>
</EsdsLiveExample>

<EsdsCodeBlock language="html">
```html
<elvt-button tone="primary">Click me</elvt-button>
```
</EsdsCodeBlock>

## Button Tones

Buttons come in four distinct tones, each serving different hierarchical purposes:

### Primary
Use for the most important actions on a page. There should typically be only one primary button visible at a time.

<EsdsLiveExample>
  <elvt-button tone="primary">Primary Action</elvt-button>
</EsdsLiveExample>

### Secondary
Use for secondary actions that are still important but not the primary call-to-action.

<EsdsLiveExample>
  <elvt-button tone="secondary">Secondary Action</elvt-button>
</EsdsLiveExample>

### Tertiary
Use for less important actions or when you need a more subtle button style.

<EsdsLiveExample>
  <elvt-button tone="tertiary">Tertiary Action</elvt-button>
</EsdsLiveExample>

### Danger
Use for destructive actions like deleting content or canceling important operations.

<EsdsLiveExample>
  <elvt-button tone="danger">Delete Item</elvt-button>
</EsdsLiveExample>

### Size Guidelines

- **Small**: Use in compact layouts, tables, or as secondary actions in cards
- **Medium**: Default size for most use cases
- **Large**: Use for prominent call-to-action buttons or on mobile interfaces

## Accessibility

The Button component is built with accessibility in mind and includes:

- **Semantic HTML**: Uses proper `<button>` elements
- **Keyboard Navigation**: Full keyboard support with proper focus management
- **Screen Reader Support**: Appropriate ARIA attributes and accessible names
- **Focus Indicators**: Clear visual focus indicators for keyboard navigation
- **Color Contrast**: All button variants meet WCAG AA contrast requirements

### Accessibility Best Practices

1. **Use descriptive text**: Button text should clearly describe the action
2. **Provide context**: For icon-only buttons, use `aria-label` attributes
3. **Manage focus**: Ensure logical focus order in forms and dialogs
4. **Handle disabled states**: Provide feedback when buttons are temporarily disabled

<EsdsCodeBlock language="html">
```html
<!-- Good: Descriptive button text -->
<elvt-button tone="primary">Save Document</elvt-button>

<!-- Good: Icon button with aria-label -->
<elvt-button tone="tertiary" aria-label="Close dialog">Ã—</elvt-button>

<!-- Avoid: Vague button text -->
<elvt-button tone="primary">Click Here</elvt-button>
```
</EsdsCodeBlock>

## Component API

### Props

| Prop | Type | Default | Description |
|------|------|---------|-------------|
| `tone` | `'primary' \| 'secondary' \| 'tertiary' \| 'danger'` | `'primary'` | Visual style and importance level |
| `size` | `'small' \| 'medium' \| 'large'` | `'medium'` | Button size variation |
| `disabled` | `boolean` | `false` | Disables button interaction |
| `type` | `'button' \| 'submit' \| 'reset'` | `'button'` | HTML button type |

### Events

| Event | Description |
|-------|-------------|
| `click` | Fired when the button is clicked |
| `focus` | Fired when the button receives focus |
| `blur` | Fired when the button loses focus |

## Guidelines

### Do's
- Use clear, action-oriented text
- Maintain consistent button hierarchy across your application
- Place primary buttons in prominent positions
- Use appropriate sizes for the context

### Don'ts
- Don't use multiple primary buttons in the same view
- Don't use buttons for navigation (use links instead)
- Don't make buttons too small for touch interfaces
- Don't use unclear or vague button labels
